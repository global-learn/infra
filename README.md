# Деплой проекта (`infra/`)

Эта папка — «пульт запуска» всего проекта на сервере. Она НЕ собирает код. Код
собирается в GitHub (см. ниже), превращается в готовые Docker-образы и кладётся
в реестр образов GitHub (GHCR). А `docker compose` из этой папки просто скачивает
эти образы и запускает их вместе с базой, redis и т.д.

## Из чего всё состоит

```
  Репозиторий бэкенда  ──push──▶  GitHub Actions собирает образ  ──▶  ghcr.io/toooweee/global-learn-cl
  Репозиторий фронта   ──push──▶  GitHub Actions собирает образ  ──▶  ghcr.io/global-learn/mvp-client
                                                                              │
                                                            на сервере: docker compose pull && up
                                                                              ▼
                            ┌──────────────── nginx (порты 80 → 443, HTTPS) ────────────────────┐
                            │   /            → client (сайт)                                     │
                            │   /api         → api (бэкенд NestJS)                               │
                            │   /socket.io   → api (вебсокеты, уведомления)                      │
                            └───────────────────────────────────────────────────────────────────┘
                                                  │
                                  api ──▶ postgres + redis + minio
```

Всё работает на одном домене по «HTTPS» (порт 443, с порта 80 — редирект на 443),
поэтому никакого CORS настраивать не нужно. «TLS-сертификат» выпускает и
автоматически продлевает Let's Encrypt (сервис `certbot`).

Поднимаемые контейнеры: `nginx` (+ `certbot`), `client`, `api`, `postgres`,
`redis`, `minio` (+ `minio-init` — разовый, создаёт бакет для файлов и сразу
завершается).

# Часть 1. Собрать образы (делается в GitHub, не на сервере)

## 1.1 Что такое «токен для пуша» и почему его НЕ надо настраивать

Когда GitHub Actions собирает образ и заливает его в реестр — он использует
**встроенный токен `GITHUB_TOKEN`**, который GitHub создаёт автоматически на время
запуска. Мы уже дали ему право `packages: write` прямо в файле workflow. **Тебе
для этого ничего настраивать не нужно** — ни секретов, ни токенов. Просто пуш.

## 1.2 Запустить сборку

В каждом из двух репозиториев есть файл `.github/workflows/release.yaml`.
Сборка запускается сама при пуше в ветку `main`:

```bash
# в репозитории бэкенда
git add .
git commit -m "ci: build & push docker image"
git push origin main

# то же самое в репозитории фронта (mvp-client)
```

Прогресс смотри на GitHub: вкладка **Actions** → workflow **«Build & Push image»**.
Когда напротив запуска появится зелёная галочка — образ готов и лежит в реестре.

## 1.3 Где появились образы

После успешной сборки на странице репозитория справа (или в профиле/организации
во вкладке **Packages**) появятся пакеты:

- бэкенд → `ghcr.io/toooweee/global-learn-cl`
- фронт → `ghcr.io/global-learn/mvp-client`

У каждого будут теги `latest` (всегда последняя сборка) и `sha-xxxxxxx`
(конкретный коммит).

# Часть 2. Дать серверу доступ к образам (вот тут про токены)

По умолчанию новые пакеты в GHCR **приватные**. Значит, чтобы сервер мог их
`docker pull`, ему нужно либо разрешение, либо пакеты надо сделать публичными.
Есть два пути — выбери один.

## Вариант А (проще всего): сделать пакеты публичными

Тогда на сервере вообще **не нужен никакой токен и `docker login`**.

Для каждого пакета (бэкенд и фронт):

1. Открой пакет на GitHub: профиль/организация → вкладка **Packages** → выбери
   нужный пакет.
2. Справа **Package settings** (⚙).
3. Внизу раздел **Danger Zone** → **Change visibility** → выбери **Public** →
   подтверди, введя имя пакета.

Готово. Можно сразу переходить к Части 3 и пропустить `docker login`.

> Подходит, если не страшно, что кто угодно сможет скачать собранный образ
> (внутри образа — твой скомпилированный код). Для учебного/дипломного проекта
> это обычно нормально.

## Вариант Б: приватные пакеты + персональный токен (PAT)

Если пакеты должны остаться приватными, серверу нужен токен на **чтение**.

### Шаг 1. Создать токен на GitHub

1. Зайди на GitHub под своим аккаунтом.
2. Открой: **Settings** (настройки профиля, не репозитория) →
   **Developer settings** → **Personal access tokens** → **Tokens (classic)**.
   Прямая ссылка: <https://github.com/settings/tokens>.
3. **Generate new token** → **Generate new token (classic)**.
4. Заполни:
   - **Note**: любое имя, например `server-pull-ghcr`.
   - **Expiration**: срок жизни (можно `90 days` или `No expiration`).
   - **Scopes** (галочки прав): поставь только **`read:packages`**.
     Больше ничего не нужно.
5. **Generate token**.
6. GitHub покажет строку вида `ghp_xxxxxxxxxxxxxxxx` — **скопируй её сразу**,
   второй раз её не покажут. Это и есть твой токен.

> Важно: токен должен принадлежать аккаунту, у которого есть доступ к обоим
> местам — к твоему профилю `toooweee` (там образ бэкенда) и к организации
> `global-learn` (там образ фронта). Раз ты владелец обоих — твой личный токен
> подойдёт.

### Шаг 2. Залогиниться этим токеном на сервере

На сервере выполни (один раз):

```bash
echo "ghp_ВСТАВЬ_СЮДА_ТОКЕН" | docker login ghcr.io -u ТВОЙ_GITHUB_ЛОГИН --password-stdin
```

Например, если логин `toooweee`:

```bash
echo "ghp_xxxxxxxxxxxxxxxx" | docker login ghcr.io -u toooweee --password-stdin
```

Должно ответить `Login Succeeded`. После этого `docker pull` из приватных
пакетов будет работать. Логин сохраняется в `~/.docker/config.json`, повторять
не нужно (пока не истечёт токен).

# Часть 3. Запустить всё на сервере

## 3.1 Что должно быть на сервере

- **Docker** и **Docker Compose v2**. Проверка:

  ```bash
  docker --version          # ожидаем 24+; должно что-то вывести
  docker compose version    # именно "compose" через пробел (v2)
  ```

  Если их нет — поставь Docker по официальной инструкции
  (<https://docs.docker.com/engine/install/>). Compose v2 ставится вместе с
  Docker как плагин.

## 3.2 Скопировать папку `infra` на сервер

Папка `infra` не в git. Перенеси её на сервер любым способом, например через `scp`
с локальной машины:

```bash
scp -r ./infra ПОЛЬЗОВАТЕЛЬ@IP_СЕРВЕРА:/opt/global-learn
```

Дальше все команды выполняются на сервере внутри этой папки:

```bash
cd /opt/global-learn   # или куда скопировал
```

## 3.3 Создать и заполнить файл `.env`

```bash
cp .env.example .env
nano .env              # или любой редактор
```

Что обязательно поменять (придумай свои значения вместо `change-me-...`):

| Переменная | Что это | Важно |
| --- | --- | --- |
| `DOMAIN` | твой домен (например `lms.example.com`) | **должен указывать на IP сервера** (см. 3.4) |
| `ACME_EMAIL` | email для Let's Encrypt | на него придут уведомления об истечении сертификата |
| `APP_URL` / `FRONTEND_URL` | публичный адрес | поставь `https://ТВОЙ_ДОМЕН` |
| `POSTGRES_PASSWORD` | пароль БД | придумай надёжный |
| `DATABASE_URL` | строка подключения к БД | **тот же пароль, что в `POSTGRES_PASSWORD`** |
| `JWT_ACCESS_SECRET` / `JWT_REFRESH_SECRET` | секреты для подписи токенов входа | случайные строки, **минимум 16 символов** |
| `MINIO_ROOT_USER` / `MINIO_ROOT_PASSWORD` | логин/пароль хранилища файлов | придумай свои |
| `TEST_ADMIN_*` / `TEST_EMPLOYEE_*` | данные стартовых аккаунтов (для сидинга) | свои email и пароли |
| `SMTP_HOST` / `SMTP_PORT` / `SMTP_FROM` | почтовый сервер для писем | укажи реальный SMTP, если нужны письма |

> Подсказка: сгенерировать случайный секрет можно так:
> `openssl rand -hex 24`

Остальное (`REDIS_IP=redis`, `MINIO_ENDPOINT=http://minio:9000`, `PORT=3000`
и т.п.) — это внутренние адреса контейнеров, их менять **не нужно**.

## 3.4 Настроить домен и выпустить HTTPS-сертификат

HTTPS работает только с **доменом** (на голый IP бесплатный сертификат не выдаётся).

1. **Направь домен на сервер.** В панели управления доменом создай `A`-запись,
   указывающую на «IP-адрес» сервера. Подожди, пока обновится «DNS» (обычно
   несколько минут). Проверить можно командой `ping ТВОЙ_ДОМЕН` — должен
   отвечать «IP» сервера.

2. **Открой порты 80 и 443** (если включён брандмауэр «UFW»):

   ```bash
   sudo ufw allow 80
   sudo ufw allow 443
   ```

3. **Впиши `DOMAIN` и `ACME_EMAIL` в `.env`** (см. таблицу выше), если ещё не.

4. **Выпусти сертификат — один раз** скриптом-помощником. Он сам поднимет
   `nginx`, получит сертификат от Let's Encrypt и перезагрузит `nginx`:

   ```bash
   chmod +x init-letsencrypt.sh
   ./init-letsencrypt.sh
   ```

   > Если хочешь сначала потренироваться, не упираясь в лимиты Let's Encrypt,
   > поставь в `.env` `CERTBOT_STAGING=1`, прогони скрипт, убедись что всё ок,
   > затем верни `CERTBOT_STAGING=0` и запусти скрипт ещё раз для настоящего
   > сертификата.

Дальше сертификат **продлевается автоматически** (этим занимается сервис
`certbot`), повторно скрипт запускать не нужно.

## 3.5 Запустить весь стек

```bash
docker compose up -d       # запускает всё в фоне
```

При старте контейнер `api` сам применит миграции базы (`prisma migrate deploy`),
поэтому отдельно ничего с БД делать не надо.

## 3.6 Проверить, что поднялось

```bash
docker compose ps          # все сервисы должны быть Up (api — после миграций)
docker compose logs -f api # смотреть логи бэкенда; Ctrl+C чтобы выйти
```

Открой в браузере:

- `https://ТВОЙ_ДОМЕН/` — сайт (фронт)
- `https://ТВОЙ_ДОМЕН/api/docs` — Swagger (документация API)

Заход по `http://` автоматически перебросит на `https://`.

## 3.7 Создать первого администратора (один раз)

В базе изначально нет пользователей. Заведи стартовые аккаунты сидом
(он возьмёт логины/пароли из `.env`, переменные `TEST_ADMIN_*` / `TEST_EMPLOYEE_*`):

```bash
docker compose exec api pnpm prisma db seed
```

# Часть 4. Обновление на новую версию

Когда ты снова запушил в `main` и GitHub собрал новый образ — на сервере:

```bash
docker compose pull && docker compose up -d
```

`pull` скачает свежий `latest`, `up -d` перезапустит изменившиеся контейнеры.
Данные БД и файлы сохранятся (они лежат в docker-томах `postgres_data` /
`minio_data`).

# Частые проблемы

- **`docker compose pull` → `denied` / `unauthorized` / `not found`.**
  Сервер не имеет доступа к приватному образу. Сделай пакеты публичными
  (Вариант А) или залогинься токеном (Вариант Б, Часть 2). Проверь, что логин
  в команде совпадает с владельцем токена.

- **`api` всё время перезапускается.** Скорее всего не поднялась БД или неверный
  `DATABASE_URL`. Посмотри `docker compose logs api` и `docker compose logs
  postgres`. Частая причина — пароль в `DATABASE_URL` не совпадает с
  `POSTGRES_PASSWORD`.

- **Сайт открывается, но запросы к API падают.** Проверь, что контейнер `api`
  в статусе `Up`, и посмотри `docker compose logs nginx` / `logs api`.

- **Файлы (аватары, обложки) не загружаются.** Проверь, что отработал
  `minio-init` (`docker compose logs minio-init` → должно быть `bucket ready`) и
  что `MINIO_BUCKET_NAME` в `.env` совпадает с тем, что ждёт приложение.

- **Письма не приходят.** В `.env` укажи рабочий `SMTP_HOST`/`SMTP_PORT`.
  По умолчанию там стоит заглушка.

- **`init-letsencrypt.sh` не получил сертификат.** Чаще всего домен ещё не
  указывает на сервер или закрыты порты. Проверь: `ping ТВОЙ_ДОМЕН` отвечает
  «IP» сервера; порты 80 и 443 открыты; `DOMAIN`/`ACME_EMAIL` заданы в `.env`.
  Логи: `docker compose logs certbot`. Если упёрся в лимит выпусков —
  поставь временно `CERTBOT_STAGING=1`.

- **Браузер ругается на сертификат.** Скорее всего в `.env` стоит
  `CERTBOT_STAGING=1` (тестовый, недоверенный сертификат). Поставь
  `CERTBOT_STAGING=0` и перевыпусти: `./init-letsencrypt.sh`.

- **`nginx` не стартует, в логах нет файла сертификата.** Сертификат ещё не
  выпущен — сначала выполни `./init-letsencrypt.sh` (шаг 3.4), потом
  `docker compose up -d`.

## Полезные команды

```bash
docker compose ps                 # статус всех сервисов
docker compose logs -f api        # логи бэкенда в реальном времени
docker compose restart api        # перезапустить один сервис
docker compose down               # остановить всё (данные сохраняются)
docker compose down -v            # остановить и УДАЛИТЬ данные (БД, файлы!) — осторожно
```

## Заметки

- HTTPS включён: `nginx` терминирует «TLS» на порту 443, а сертификат Let's
  Encrypt выпускается скриптом `init-letsencrypt.sh` и автоматически продлевается
  сервисом `certbot`. Никаких ручных действий с сертификатом после первого
  запуска не требуется.
- Консоль MinIO наружу не открыта. Если нужна — добавь сервису `minio` проброс
  порта `"9090:9090"`.
- Чтобы зафиксировать конкретную версию вместо `latest`, поменяй теги в
  `docker-compose.yaml` на `...:sha-<короткий-хеш>` (такие теги тоже публикуются).
